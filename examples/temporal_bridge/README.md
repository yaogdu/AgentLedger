# Temporal Bridge Retry Safety Demo

This dependency-free example shows the intended boundary between a Temporal-style durable workflow backend and AgentLedger.

```text
Temporal-style workflow/activity facade
  -> retries an activity after a worker crash
  -> AgentLedger runtime inside the activity
  -> ToolGateway / Tool Ledger / evidence
  -> external side effect
```

Temporal owns the outer workflow lifecycle, scheduling, and activity retry policy. AgentLedger owns the node-internal model/tool/state reliability boundary.

Run from the repository root:

```bash
PYTHONPATH=src python3 examples/temporal_bridge/demo.py
```

Expected behavior:

- the first activity attempt starts an AgentLedger run
- a runtime-managed external-write tool succeeds
- the activity crashes before framework/workflow-level success
- the Temporal-style facade retries the activity and reuses archived proposal evidence
- AgentLedger reuses the Tool Ledger result and does not repeat the external side effect
- the output includes workflow/activity correlation metadata, two activity attempts, one external action, replay summary, failure summary, evidence directory, and Inspector HTML path

This is not a Temporal SDK dependency and does not require a Temporal server. In a production adapter, replace the local facade with a real Temporal workflow/activity and pass workflow/activity ids into the AgentLedger run state or event metadata.
