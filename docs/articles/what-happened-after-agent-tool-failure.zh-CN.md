# Agent 调用工具失败后，到底发生了什么？

很多 Agent demo 能跑通工具调用，但生产里更麻烦的问题通常不是“工具能不能调”，而是：

- 工具真的执行了吗？
- 执行成功后 worker 崩了怎么办？
- 重试会不会重复发邮件、创建 ticket、改数据库？
- 失败后有没有证据能审计和复盘？
- replay 时会不会再次触发副作用？

Agent 框架通常负责 planning、reasoning 和 workflow logic。AgentLedger 关注的是另一个边界：Agent 和外部世界发生交互之后，runtime 如何记录、恢复、审计和 replay。

## 一个最小 crash 场景

假设 Agent 要发送一封邮件：

```text
1. Agent 调用 email.send
2. 邮件已经发送成功
3. worker 在写 checkpoint 前崩溃
4. 调度系统重试这个任务
```

如果没有 runtime ledger，重试时 Agent 很可能再次调用 `email.send`。结果是用户收到两封邮件，系统里却没有可靠证据说明第一次到底发生过什么。

## AgentLedger 做了什么

AgentLedger 在 runtime boundary 上记录副作用：

- Tool Ledger：记录工具调用、请求 hash、idempotency key、状态和外部结果
- checkpoint：只在状态提交成功后推进 run state
- failure evidence：记录 crash、retry、失败分类和上下文
- replay：用证据复盘执行路径，不重新调用副作用工具
- Inspector：导出只读 HTML，查看 run timeline、Tool Ledger、failure 和 JSON 详情

也就是说，retry 不再只是“再跑一遍”。retry 会先经过 runtime 记录，识别已经发生过的副作用，并复用已有结果。

你会看到两个对比：

```text
WITHOUT runtime ledger

  actual_tool_executions: 1
  replay_safe: true
```

demo 还会生成：

- `runs.html`
- `inspector.html`

打开 HTML 可以看到 Tool Ledger、failure/retry evidence、run timeline 和 replay-safe evidence。

## 这个项目解决什么问题

AgentLedger 不是新的 Agent 框架，也不是模型网关。它是面向生产 Agent 的 runtime reliability layer。

它适合这些场景：

- Agent 会调用有副作用的工具
- 任务可能长时间运行或被重试
- 你需要恢复、审计、debug、replay
- 你需要证明某次 tool call、model call 或 failure 到底发生了什么
- 你希望和 LangGraph、Temporal、Langfuse、MCP、OpenAI Agents SDK 等生态组合使用
